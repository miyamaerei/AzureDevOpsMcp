# 代码提交规范

## 1. 概述

本文档定义了项目的代码提交规范，旨在提高代码提交的可读性和可追溯性，便于团队协作和问题定位。

## 2. 提交格式

### 2.1 基本格式

```
<类型>(<范围>): <主题>

<正文>

<页脚>
```

### 2.2 格式说明

| 部分 | 要求 | 说明 |
|------|------|------|
| **类型** | 必填 | 提交的类型，如 feat、fix、docs 等 |
| **范围** | 可选 | 提交影响的模块或文件 |
| **主题** | 必填 | 简短描述提交内容，不超过 50 字符 |
| **正文** | 可选 | 详细描述提交的原因和变更内容 |
| **页脚** | 可选 | 引用相关 Issue 或 PR |

## 3. 提交类型

### 3.1 类型定义

| 类型 | 说明 | 示例 |
|------|------|------|
| **feat** | 新增功能 | `feat(task): 添加状态流转验证服务` |
| **fix** | 修复 Bug | `fix(api): 修复状态映射错误` |
| **docs** | 文档更新 | `docs: 更新开发流程文档` |
| **style** | 代码格式调整（不影响功能） | `style: 格式化代码` |
| **refactor** | 代码重构（不改变功能） | `refactor: 重构任务同步服务` |
| **test** | 测试相关 | `test: 添加状态流转测试用例` |
| **chore** | 构建/工具相关 | `chore: 更新依赖版本` |
| **perf** | 性能优化 | `perf: 优化数据库查询` |
| **revert** | 回滚提交 | `revert: 撤销上次提交` |

### 3.2 类型选择指南

1. **feat**: 当添加新的功能、API、工具方法时使用
2. **fix**: 当修复 bug、错误或异常时使用
3. **docs**: 当修改或添加文档时使用
4. **style**: 当只修改代码格式（如缩进、空格、换行）时使用
5. **refactor**: 当重构代码结构但不改变功能时使用
6. **test**: 当添加或修改测试用例时使用
7. **chore**: 当修改构建脚本、配置文件或依赖时使用
8. **perf**: 当优化性能时使用
9. **revert**: 当撤销之前的提交时使用

## 4. 范围定义

范围用于标识提交影响的模块或文件，建议使用以下值：

| 范围 | 说明 |
|------|------|
| `api` | API 接口相关 |
| `auth` | 认证授权相关 |
| `config` | 配置相关 |
| `docs` | 文档相关 |
| `model` | 数据模型相关 |
| `service` | 服务层相关 |
| `tool` | MCP 工具相关 |
| `test` | 测试相关 |
| `workflow` | 工作流相关 |

## 5. 主题规范

### 5.1 要求

- 不超过 50 个字符
- 使用祈使句（动词开头）
- 首字母大写
- 不使用句号结尾

### 5.2 示例

✅ 正确：
- `feat(service): 添加状态流转验证服务`
- `fix(tool): 修复任务同步逻辑`
- `docs: 更新开发流程文档`

❌ 错误：
- `add state transition service`（缺少类型）
- `Feat: 添加服务`（范围缺失，风格不一致）
- `fix bug in task sync`（描述不够具体）

## 6. 正文规范

### 6.1 要求

- 详细描述提交的原因和变更内容
- 每行不超过 72 字符
- 使用 Markdown 格式

### 6.2 示例

```
feat(service): 添加状态流转验证服务

- 实现 ITaskStatusTransitionService 接口
- 添加状态流转矩阵和描述字典
- 支持四种状态之间的合法转换
- 提供状态转换验证和描述获取功能

Closes #123
```

## 7. 页脚规范

### 7.1 Issue 引用

使用 `Closes`、`Fixes` 或 `Resolves` 引用相关 Issue：

```
Closes #123
Fixes #456
Resolves #789
```

### 7.2 Breaking Change

当提交包含破坏性变更时，使用 `BREAKING CHANGE:` 标识：

```
refactor(api): 修改任务状态枚举命名

BREAKING CHANGE: 将 TaskStatus 枚举中的 NotStarted 改为 NotImplemented，InProgress 改为 Current

请更新所有引用该枚举的代码。
```

## 8. 提交示例

### 8.1 功能添加

```
feat(service): 添加状态流转验证服务

实现了基于 PRD 定义的状态流转规则验证服务：
- 支持四状态任务模型
- 实现状态流转矩阵
- 提供转换验证和描述功能

Closes #1
```

### 8.2 Bug 修复

```
fix(tool): 修复任务同步状态映射错误

修复了从 Azure DevOps 拉取任务时状态映射不正确的问题：
- 将 NotStarted 改为 NotImplemented
- 将 InProgress 改为 Current

Fixes #2
```

### 8.3 文档更新

```
docs: 更新开发流程文档

- 添加四状态任务模型说明
- 添加状态流转规则表
- 添加任务生命周期流程图
```

### 8.4 代码重构

```
refactor(service): 重构任务同步服务

- 分离状态转换逻辑到独立服务
- 优化同步流程
- 提高代码可测试性

BREAKING CHANGE: ITaskSyncService 接口签名变更
```

## 9. 提交检查清单

在提交前，请检查：

- [ ] 提交类型正确
- [ ] 主题简洁明了（≤ 50 字符）
- [ ] 使用祈使句
- [ ] 正文清晰描述变更原因和内容
- [ ] 引用相关 Issue
- [ ] 如果有破坏性变更，添加 BREAKING CHANGE 说明
- [ ] 所有测试通过

## 10. 工具支持

推荐使用以下工具辅助遵守规范：

- **commitlint**: 提交消息校验
- **cz-cli**: 交互式提交消息生成
- **pre-commit**: 提交前钩子
